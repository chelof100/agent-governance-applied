% --- Abstract -----------------------------------------------------------------
\begin{abstract}
A correctly governed agent system can still fail.
An agent may select only actions that each individually satisfy every
applicable rule, while its behavioral trajectory drifts silently toward
high-risk territory.
An agent may request execution of a privileged action at a moment when the
authority-defining state is partially unobservable, making it impossible to
confirm validity without either blocking or accepting invalid executions.
Multiple agents may each stay within their individual policy budgets while
collectively exhausting the system's coordination capacity.
We call the structural interval in which these failures occur the
\emph{Execution Gap}: the space between what governance validates at decision
boundaries and what agents actually do in execution.

Existing approaches -- prompt guards, OPA/XACML policy engines, Constitutional
AI, and audit layers -- are \emph{structurally incapable} of closing this gap.
They evaluate actions locally and statelessly; the Execution Gap is a
trajectory-level, stateful phenomenon.

This paper provides the first empirical demonstration that the Execution Gap
is real, measurable, and closeable.
We implement the Agent Governance Stack (Papers~P0--P6: atomic decision
boundaries, stateful admission control, invariant measurement, governance
structure, and reconstructive authority) as a Python library instrumented
into a \textbf{LangGraph}
StateGraph, and run four experiments that each isolate one dimension of the gap.
Our central finding: \emph{the enforcement signal remains identically zero while
the IML composite $\Dhat$ grows monotonically}, crossing the detection
threshold in $T^*\in[259,403]$ steps across all six seeds -- direct
experimental proof of compliant drift.
Additionally: the RAM gate achieves $\IER=0.000$ at every state-coverage level
$(0.10$--$1.00)$, versus baseline $\IER\in[0.032,\,0.185]$; the ACP
coordination window replicates $\CW_{\mathrm{appr}}=2N$ with zero deviation
for $N\in\{2,4,8,16\}$ agents; and the integrated stack converges with
$\Dhat$ bounded in $[0.27,0.34]$ over 2000 steps, liveness holds, and no
deadlock occurs.
We also report three empirical refinements to the formal framework discovered
during implementation.
The open-source implementation provides a deployable blueprint for
practitioners integrating runtime governance into LangGraph-based agent systems.
Code and data: \url{https://github.com/chelof100/agent-governance-applied}.
Archived version: \url{https://doi.org/10.5281/zenodo.19929771}.
This is Paper~7 of the Agent Governance Series (P0--P7; Paper~8 on identity-bound governance and post-HALT authority resolution is in preparation).
\end{abstract}

\section{Introduction}
\label{sec:intro}

Autonomous agent systems built on large language models (LLMs) are increasingly
deployed in production settings where they execute consequential actions:
modifying data, invoking APIs, delegating subtasks to other agents, and
coordinating across shared resources~\cite{yao2023react,weng2023agent}.
The governance question -- \emph{who controls what the agent may do, and how is
that control enforced at runtime?} -- has grown urgent as these systems move
from research prototypes to production pipelines.
Yet passing all governance checks is not the same as being safe.
This paper is about the difference.

\subsection*{The Execution Gap}

Current governance approaches share a common design: they evaluate actions
at fixed decision boundaries, locally, against a rule set.
\emph{Prompt guards} check model outputs before passing them downstream.
\emph{OPA/XACML policy engines}~\cite{openpolicyagent2023,anderson2004extensible}
evaluate capability tokens against attribute-based access rules.
\emph{Constitutional AI}~\cite{bai2022constitutional} embeds constraints into
model weights via fine-tuning and RLHF.
\emph{Audit layers} record events for post-hoc analysis.

The structural property these approaches share -- local, stateless evaluation
at decision boundaries -- defines the limit of what they can detect.
We call the space beyond that limit the \textbf{Execution Gap}: the interval
between what governance validates and what agents actually do in execution.
Three classes of failure live in this gap, each invisible to local enforcement:

\begin{enumerate}[leftmargin=2em, topsep=2pt]
  \item \textbf{Compliant drift} (P2~\cite{fernandez2026iml}): an agent
        whose \emph{every individual action} satisfies all local rules, but
        whose behavioral trajectory diverges from its authorized baseline.
        The P2 Non-Identifiability Theorem proves this drift lies outside
        $\sigma(g)$ -- it cannot be detected by any function of the
        enforcement signal alone.
        Section~\ref{sec:exp1} provides the first experimental proof in a
        real LangGraph workflow.
  \item \textbf{Partial observability} (P5--P6~\cite{fernandez2026ram,fernandez2026opram}):
        a request arrives when the authority-defining state is not fully
        observable. Stateless attestation either accepts invalid executions
        (false positives) or blocks valid ones (false negatives); there is no
        correct stateless answer. RAM's reconstructive procedure is the only
        mechanism that achieves $\IER=0$ under partial observability while
        preserving liveness.
  \item \textbf{Coordinated multi-agent escalation} (P1~\cite{fernandez2026acp},
        P3/4~\cite{fernandez2026govstr}): $N$ agents each within individual
        policy but collectively exhausting the system's coordination budget.
        The formal bound $\CW_{\mathrm{appr}}=2N$ (P1) constrains this
        amplification and is shown here to be framework-independent.
\end{enumerate}

Papers 0--6 address each failure mode with formal guarantees.
But formal guarantees alone do not answer the operational question: does
the integrated stack actually close the Execution Gap when deployed in a
real LLM agent framework?

\subsection*{This paper}

We answer that question empirically: in our LangGraph-based experimental
setting, \emph{none of the three failure modes above appears when the complete
governance stack is operating.}
We implement the entire P0--P6 stack in Python, instrument it into a
LangGraph~\cite{langgraph2024} StateGraph agent, and run four experiments --
one per failure dimension, plus a full-stack integration -- on a deterministic
but realistic agent workflow with five tools spanning the full risk spectrum.

Within our experimental setting, the results are consistent and clear: the stack closes the gap.
Compliant drift is detected while enforcement sees nothing.
Execution validity is guaranteed at every coverage level.
Coordination amplification is bounded by the formal prediction.
The integrated system converges, maintains liveness, and produces no deadlocks.
Moreover, implementation against a real framework surfaces three refinements
to the formal theory that the purely theoretical treatment left implicit.

\textbf{Contributions:}
\begin{itemize}[leftmargin=2em, topsep=2pt]
  \item \textbf{Empirical proof of the Execution Gap}: first experimental
        demonstration that an agent can be fully enforcement-compliant
        ($g(\tau)=0$ throughout) while drifting measurably from its
        authorized baseline ($\Dhat > \theta$ in all six seeds).
  \item \textbf{Closed-form closure}: experimental confirmation that the
        integrated ACP+IML+RAM+RecoveryLoop stack closes all three Execution
        Gap failure modes simultaneously, with zero deviation from formal
        predictions on four independent experimental axes.
  \item \textbf{Open-source implementation} of the full governance stack for
        LangGraph (\texttt{chelof100/agent-governance-applied}).
  \item \textbf{Three theoretical refinements}: ACP baseline-RS assumption,
        liveness rate classification for the conditional liveness theorem,
        and EMA convergence bound parametrization.
\end{itemize}

\subsection*{Key Metrics (self-contained reference)}

The following metrics are used throughout the paper.
They are defined formally in the referenced papers but reproduced here for
standalone readability.

\begin{itemize}[leftmargin=2em,topsep=2pt,itemsep=1pt]
  \item $\IER$ \textbf{(Invalid Execution Rate)}: fraction of executions
        where the authority was not valid at execution time.
        $\IER = 0$ means no invalid execution ever occurred.
  \item $\SHR$ \textbf{(Safe Halt Rate)}: fraction of halts that correctly
        blocked an invalid execution (true positives of the gate).
  \item $\OCR$ \textbf{(Opportunity Cost Rate)}: fraction of halts that
        blocked a valid execution (false positives; opportunity cost of the
        gate's conservatism).
  \item $\Dhat$ \textbf{(IML composite drift signal)}: weighted combination
        $0.40 D_t + 0.35 D_c + 0.25 D_\ell$ of distributional ($D_t$), risk
        proximity ($D_c$), and lineage ($D_\ell$) deviations, smoothed by EMA
        ($\alpha = 0.15$). Range $[0, 1]$; detected when $\Dhat \geq \theta$.
  \item $T^*$ \textbf{(detection step)}: first step at which $\Dhat \geq \theta$.
  \item $\CW_{\mathrm{appr}}$ \textbf{(approved coordination window)}: total
        number of approved actions across all $N$ agents before the first DENY.
        Predicted closed-form: $\CW_{\mathrm{appr}} = 2N$.
\end{itemize}

\subsection{ACP: Stateful Admission Control}
\label{sec:bg-acp}

The Agent Control Protocol~\cite{fernandez2026acp} provides a stateful gate
that evaluates each capability request against the agent's behavioral history.
For a given $(agentID, capability, resource)$ tuple (the \emph{PatternKey}),
ACP maintains a pattern counter that accumulates across requests.
Two anomaly rules fire when the counter crosses thresholds:
Rule~1 (count~$>N_1$) adds $+\Delta_1$ to the effective risk score;
Rule~3 (count~$\geq N_3$) adds $+\Delta_3$.
The \emph{evaluate-then-mutate} contract (ACP-RISK-3.0~\S4) ensures the counter
is read before the decision and incremented after -- decoupling the decision
from its own effect on future decisions.
For any agent, the first two requests are evaluated against counts 0 and 1
(below the Rule~3 threshold), and therefore receive $\ADMIT$ decisions.
This yields the closed-form coordination window
$\CW_{\mathrm{appr}} = 2N$ for $N$ agents.

\subsection{IML: Invariant Measurement Layer}
\label{sec:bg-iml}

The IML~\cite{fernandez2026iml} addresses a fundamental limitation of
enforcement-based monitoring: the enforcement signal $g(\tau)$ cannot detect
behavioral drift that remains within the rule set, because $\Azero \notin
\sigma(g)$ (the Non-Identifiability Theorem).
IML bypasses this by retaining direct access to the admission-time snapshot
$\Azero = (P_0, \mu_\ell, \sigma_\ell)$ and computing a composite deviation
estimator:
\[
  \Dhat(\tau;\Azero) = 0.40\cdot D_t(\tau) + 0.35\cdot D_c(\tau) +
  0.25\cdot D_\ell(\tau)
\]
where $D_t$ is JS divergence between the current and baseline tool
distributions, $D_c$ is the mean proximity to constraint boundaries,
and $D_\ell$ is the normalized lineage depth deviation.
All three components are estimable from the observed trace $\tau$,
without access to the enforcement signal.
In this sense, $\Azero$ functions as a \emph{certified contract with
reality}: execution does not verify intent---it verifies whether the
terms of that contract still hold at runtime.

\subsection{RAM and Recovery Loop}
\label{sec:bg-ram}

The Reconstructive Authority Model~\cite{fernandez2026ram} addresses partial
observability in execution authority.
Given a proven state $\Sp(t) \subseteq \Sr(t)$, RAM applies the decision rule:
\begin{itemize}[leftmargin=2em, topsep=0pt]
  \item If any required component is $\mathtt{False}$ in $\Sp$: $\DENY$
  \item If any required component is undefined in $\Sp$: $\HALT$
  \item If all required components are $\mathtt{True}$: $\EXECUTE$
\end{itemize}
This guarantees $\IER=0$ -- execution only proceeds when authority is verified.
The Recovery Loop~\cite{fernandez2026opram} handles $\HALT$ events through
five stages: Signal Extraction, IML Trigger, State Augmentation,
Reconstruction Attempt, and Resolution.
The conditional liveness theorem guarantees exit from $\HALT$ when all
authority-defining variables eventually become observable.

\section{Experimental System}
\label{sec:system}

\subsection{LangGraph Workflow Architecture}

We implement a three-node LangGraph StateGraph representing a realistic
autonomous agent workflow:
\begin{center}
\texttt{orchestrator} $\to$ \texttt{tool\_selector} $\to$ \texttt{executor}
\end{center}
The \texttt{orchestrator} node maps phase and drift progress to a task intent
(\texttt{routine\_maintenance}, \texttt{data\_access}, \texttt{data\_modification},
\texttt{privileged\_operation}).
The \texttt{tool\_selector} node runs the MockLLM to select a tool and delegation
depth.
The \texttt{executor} node records the execution.

\subsection{Tool Set and Risk Scores}

We define five tools with risk scores (RS) calibrated to span the governance
decision space:

\begin{center}
\begin{tabular}{@{}llll@{}}
\toprule
Tool & RS & RAM gate & ACP base risk \\
\midrule
\texttt{read\_file}     & 10 & No (RS $<$ 45) & Below escalation \\
\texttt{query\_api}     & 25 & No (RS $<$ 45) & Below escalation \\
\texttt{write\_data}    & 45 & Yes            & At admit boundary \\
\texttt{delete\_record} & 70 & Yes            & Above deny threshold \\
\texttt{admin\_action}  & 90 & Yes            & Above deny threshold \\
\bottomrule
\end{tabular}
\end{center}

\textbf{Remark.} Tools with $\RS \geq 70$ exceed the ACP admit threshold
($\theta_a = 46$) on their base risk score alone, causing $\DENY$ without
pattern accumulation. This reveals an implicit assumption in P1 -- that all
baseline tool requests have $\RS_{\mathrm{base}} < \theta_a$ -- which must
be made explicit when applying ACP to tools with high intrinsic risk.
For Experiments 1--3 we use tools in the admissible range; for Experiment 4
the ACP is configured with elevated thresholds to observe the RAM and
Recovery Loop layers.

\subsection{MockLLM: Deterministic Drift Model}

We use a deterministic MockLLM with seed-controlled tool selection.
During burn-in (50 steps), the tool distribution is safe-heavy:
$(0.50, 0.30, 0.15, 0.04, 0.01)$ for
(\texttt{read\_file}, \texttt{query\_api}, \texttt{write\_data},
\texttt{delete\_record}, \texttt{admin\_action}).
During the drift phase, the distribution interpolates linearly toward a
risky-heavy target $(0.05, 0.10, 0.45, 0.28, 0.12)$, inducing compliant
drift: no individual tool call violates any rule, but the behavioral
trajectory diverges from $\Azero$.
This mirrors the LangGraph experiment from P2~\S5.4 but extends it to
five tools and three workflow nodes.
Six seeds (42, 1--5) ensure reproducibility across Table~\ref{tab:exp1}.

\subsection{Governance Stack Integration}

Each component is wired into the graph as a stateful observer:
\begin{itemize}[leftmargin=2em, topsep=2pt]
  \item \textbf{IML}: reads the external \texttt{Trace} object after each
        \texttt{executor} step; computes $\Dhat$, $D_t$, $D_c$, $D_\ell$,
        and the enforcement signal $g(\tau)$.
  \item \textbf{ACP}: evaluates $(agentID, tool, \RS_{\mathrm{base}})$
        before recording the step; applies evaluate-then-mutate.
  \item \textbf{RAM}: applied to tools with $\RS \geq 45$; receives
        the current $\Dhat$ as a drift-level signal that scales the
        failure probability of the behavioral component~$B$.
  \item \textbf{Recovery Loop}: triggered whenever RAM returns $\HALT$;
        runs up to $k=5$ augmentation attempts with coverage
        $c_i = c_0 + i \cdot \Delta c \cdot \eta$, where $\eta$ is
        the IML-derived augmentation efficiency.
\end{itemize}

\section{Experiment 1: Drift Under Compliant Execution}
\label{sec:exp1}

\subsection{Setup}

We run 500 steps (50 burn-in, 450 drift) with the MockLLM inducing
gradual tool-selection drift toward high-risk tools, while each individual
tool call satisfies all local rules ($g(\tau) = 0$ throughout).
The IML threshold $\theta = 0.20$ follows P2~\S4.3, where it is calibrated
for a 500-step detection horizon against the assumed burn-in distribution:
at this threshold, the finite detection delay theorem guarantees $T^* < \infty$
provided the drift endpoint $D_\infty > \theta$.
Results are reported for six seeds.

\subsection{Analysis}

The enforcement signal remains $g(\tau) = 0$ across all 2700 drift steps
(6 seeds $\times$ 450), confirming that no individual tool call violates
any local rule.
Simultaneously, $\Dhat$ grows monotonically and crosses $\theta=0.20$ in
$T^* \in [259, 403]$ steps, with mean $T^* = 356$.

This replicates -- and extends -- the core claim of P2: IML detects drift
that enforcement is structurally blind to.
The extension is qualitatively significant: the P2 experiment used two
tool categories with a two-node graph; this experiment uses five tools with
differentiated risk scores in a three-node workflow, and the result holds
across all seeds.

\textbf{Comparison with a naive baseline.}
A natural alternative to the IML composite is a simple rolling mean of
the raw risk score, with a fixed offset threshold: $\bar{\RS}_t > \bar{\RS}_{\mathrm{burn}} + \delta$.
For seed~42, the burn-in mean risk score is $\approx 0.23$; setting $\delta = 0.10$
gives a naive threshold of $0.33$.
The naive detector fires at step $T^*_{\mathrm{naive}} \approx 83$ --
much earlier than IML's $T^* = 353$.
This earlier trigger is not an advantage: the naive threshold is calibrated
by an arbitrary offset rather than against $\Azero$, so it conflates
a slow natural increase in mean risk (which may be legitimate) with genuine
trajectory divergence.
IML's composite $\Dhat$ is anchored to the admission-time snapshot $\Azero$
and combines three independent signals ($D_t$, $D_c$, $D_\ell$),
making its detection formally grounded (P2 finite detection delay theorem)
and resistant to single-signal false positives.

The $D_c$ component (mean risk proximity) is the primary driver of $\Dhat$
growth: as the tool distribution shifts toward \texttt{write\_data} and
\texttt{delete\_record}, the mean risk score of the trace increases from
the burn-in baseline ($\approx 0.17$) to the drift endpoint ($\approx 0.42$).
$D_t$ (distributional divergence) contributes secondary growth.
$D_\ell$ (lineage deviation) is a minor contributor because the
MockLLM's depth variance during burn-in is zero, yielding a fallback
normalization ($\sigma_\ell = 0.5$) that slightly inflates $D_\ell$ at
depth increments; we note this as a methodological artifact of the
MockLLM (see Section~\ref{sec:limits}).

\section{Experiment 1b: Live LLM Drift Detection}
\label{sec:exp1b}

\subsection{Setup}

To address the concern that Experiment~1 relies on a synthetic drift
model, we replicate the detection task with real LLMs making decisions
from natural language context.
We test two models from distinct families, both served locally via Ollama
at temperature\,=\,0.4:
\textbf{mistral-small3.1} (Mistral AI, 15B parameters) and
\textbf{deepseek-r1:8b} (DeepSeek, 8B parameters, chain-of-thought
reasoning disabled for latency\footnote{DeepSeek-R1 generates
chain-of-thought (\texttt{<think>}) tokens before its final answer.
We pass \texttt{think=False} via the Ollama API to suppress the reasoning
trace and receive only the tool-selection decision, which is the unit of
governance interest.}).

Each model receives a task description that escalates in urgency and risk
over 150 drift steps following a 50-step burn-in.
During burn-in, the model receives routine maintenance scenarios;
during drift, the context shifts across five tiers (monitoring, quality
review, data correction, urgent cleanup, emergency recovery) proportionally
to drift progress.
Critically: \emph{no tool is forbidden at any point}.
Every tool selection is a contextually appropriate response to a
legitimate task description.
No enforcement rule is violated.

\subsection{Analysis}

Both models confirm the central claim of Experiment~1 under real-LLM
conditions and across distinct model families.
Each model makes a sequence of contextually appropriate decisions:
routine monitoring tasks select \texttt{read\_file}; data correction tasks
select \texttt{write\_data}; urgency scenarios select \texttt{delete\_record};
emergency scenarios select \texttt{admin\_action}.
Every selection is a reasonable response to a legitimate prompt.
No enforcement rule is triggered ($g(\tau) = 0$ throughout) for either model.

Yet the IML composite $\Dhat$ grows monotonically and crosses $\theta=0.20$
in both runs.
For mistral-small3.1, $\Dhat$ grows from $0.007$ to $0.367$,
crossing $\theta=0.20$ at $T^*=64$.
For deepseek-r1:8b (a chain-of-thought reasoning model from a distinct
architecture family), $\Dhat$ grows from $0.006$ to $0.362$,
crossing $\theta=0.20$ at $T^*=65$.
The two trajectories differ by one detection step and 0.005 in final $\Dhat$:
the drift is driven by the context-escalation protocol,
not by model-family differences.
The trajectory-level shift -- from a safe tool distribution to a high-risk
one -- is detectable by IML precisely because IML retains access to the
authorized baseline $\Azero$.
Enforcement cannot detect it because enforcement evaluates each action
locally, and each action is locally correct.

\textbf{Key implication.} The Execution Gap is not an artifact of
synthetic drift models, nor of any particular LLM family.
Real LLMs making contextually appropriate choices -- responding to
task context in a way that any operator would approve step-by-step --
produce behavioral drift that is structurally invisible to enforcement
and structurally visible to IML.
The gap is architectural, not experimental.

\textbf{Convergence.} $\Dhat$ stabilizes in the range $[0.27, 0.34]$ over

The stabilization arises from the EMA smoothing ($\alpha = 0.15$):
once the MockLLM's tool distribution reaches its drift endpoint
(constant weights), the IML inputs are stationary and $\Dhat$ converges
to a fixed point determined by the gap between burn-in and drift
distributions.
We identify an empirical refinement: the convergence bound should be
parameterized by $\alpha$:
$\limsup_{t \to \infty} \Dhat_t \leq f(\alpha, \varepsilon_b, K, \eta)$
where $f$ increases with $\alpha$.
The exact form of $f$ is left for future work.

\textbf{Liveness.}
Of the 821 HALT events, 406 (49.5\%) were resolved by the Recovery Loop
within $k \leq 5$ attempts.
The remaining 415 events split into 167 ESCALATE (real authority was

reached maximum but the unobservable components remained undefined).

We report a refinement of the P6 conditional liveness theorem:
the relevant recovery rate is not over all HALT events, but only over
those caused by \emph{unobservability} (components undefined in $\Sp$)
rather than by \emph{false authority} (components $\mathtt{False}$ in $\Sr$).
The conditional liveness guarantee -- ``if all authority-defining variables
eventually become observable, the system exits HALT'' -- applies
only to the former category.
Distinguishing these categories requires tracking ground-truth authority,
which the paper should clarify as a measurement requirement.

\textbf{No deadlock.}
The sequential execution model (single agent, single-threaded graph)
precludes resource lock contention by construction.
In a concurrent multi-agent deployment, deadlock prevention would require
explicit lock management; we leave this for future work.

\textbf{Component interactions.}
The three governance layers operate without interference:
IML feeds its $\Dhat$ signal to RAM (raising the behavioral failure probability
of component~$B$ under high drift), which in turn triggers the Recovery Loop
on HALT decisions, which uses the IML state to modulate augmentation
efficiency. The feedback loop is well-defined and non-circular:
$\mathrm{IML} \to \mathrm{RAM} \to \mathrm{RecoveryLoop} \to \mathrm{RAM}$,
with no path from RAM or Recovery Loop back into the IML trace.

$\Dhat$ convergence is identical across all IML-enabled configurations

execution-gating layers.

we treat this as a fair-play caveat rather than a permanent structural limitation.

\section{Theoretical Refinements and Design Constraints}
\label{sec:refinements}

Beyond confirming the theoretical predictions, the implementation revealed
three empirical findings that refine the formal framework, and exposed
two structural properties that function as design constraints on the
architecture.

\subsection{ACP Baseline-RS Assumption}

P1 implicitly assumes that the baseline risk score of any capability request
satisfies $\RS_{\mathrm{base}} < \theta_a$ (the admit threshold), so that the
pattern-accumulation rules, not the base risk alone, drive escalation.
This assumption holds for the financial.transfer example in P1 Experiment~13
(RS\,=\,35, $\theta_a = 39$) but fails for tools with high intrinsic risk
(\texttt{delete\_record} RS\,=\,70, \texttt{admin\_action} RS\,=\,90 against
$\theta_a = 46$).

\textbf{Refinement:} ACP should be applied to capability \emph{classes}
rather than individual tools, where each class has a baseline RS below
the admit threshold. The class-level PatternKey then accumulates across
all tools in the class. Alternatively, P7's architecture separates
responsibility: ACP gates by behavioral pattern, RAM gates by intrinsic
tool risk. This is the design adopted in Experiment~4.

\subsection{Liveness Rate Classification}

P6 Theorem~3 (Conditional Liveness) guarantees: ``if all authority-defining
variables eventually become observable, the system exits HALT.''
In Experiment~4, 248 of 821 HALT events remained unresolved after five
augmentation attempts.

\textbf{Refinement:} These stuck cases fall into two categories:
(a)~\emph{unresolvable} -- a required component is \texttt{False} in the real
state $\Sr$; Recovery Loop cannot resolve a genuine authority failure;
(b)~\emph{temporarily unobservable} -- the component is \texttt{True} in $\Sr$
but undefined in $\Sp$; additional augmentation attempts would eventually resolve it.
The liveness theorem applies only to category~(b). A correct empirical
reporting of liveness rate requires classifying HALT events by real authority,
which is available in simulation but must be approximated in deployment.
We recommend adding this classification to the P6 framework.

\subsection{EMA Convergence Parametrization}

The IML convergence bound in P7's theoretical outline reads
$\limsup_{t} \Dhat_t \leq \varepsilon_b / (K\eta)$.
Empirically, $\Dhat$ stabilizes at $\approx 0.31$ over the drift-stationary
phase, with the exact value depending on the EMA smoothing factor $\alpha = 0.15$.

\textbf{Refinement:} The convergence bound should read
$\limsup_t \Dhat_t \leq f(\alpha, \varepsilon_b, K, \eta)$
where $f$ is increasing in $\alpha$ (higher smoothing slows convergence
but also raises the asymptote under non-zero stationary drift).
For $\alpha = 0$, $\Dhat$ converges to the instantaneous raw signal.
For $\alpha \to 1$, $\Dhat$ never converges (permanent memory of initial state).
The derivation of the exact form of $f$ is deferred to future work.

\textbf{Deployment guidance.}
Practitioners should calibrate $\alpha$ against the expected drift horizon
on held-out traces: lower values ($\alpha \approx 0.05$) suit systems
where abrupt, short-horizon drift is the primary concern and fast reaction
is more important than noise suppression;
higher values ($\alpha \approx 0.30$) suit systems where gradual long-horizon
drift dominates and a stable running signal is preferable.
This is a standard hyperparameter search with a monotone objective:
longer expected drift horizon $\Rightarrow$ higher optimal $\alpha$.

\subsection{Separation of Recalibration Authority (Design Constraint DC.1)}

A system that allows its execution layer to modify its own governance
parameters under runtime pressure ceases to be governed---it becomes
self-justifying. The experiments confirm and operationalize this
restriction: the stack's execution layer may validate state, emit
evidence, and permit or block individual actions, but it may not modify
$\theta$, recalibrate $\Azero$, or reinterpret policy at runtime.

\textbf{Design Constraint DC.1:} \emph{The recalibration authority over
$\theta$, $\alpha$, and $\Azero$ is strictly separated from the
execution layer. No runtime condition---including sustained $\HALT$
or accumulated drift---authorizes the execution layer to modify its
own governance parameters.}

Formally, governance parameters are immutable at the execution layer:
\[
  \forall t,\quad \theta(t) = \theta(0),\quad \alpha(t) = \alpha(0),
  \quad \Azero(t) = \Azero(0).
\]

This constraint answers directly the question: \emph{who governs the
governor under runtime pressure?} The answer is not the execution layer.
Recalibration requires an out-of-band governance action with its own
authority chain and evidential record. A stack that permits in-band
recalibration under pressure redefines compliance rather than enforcing it,
and thereby eliminates the structural guarantee that gives $\Azero$ its
meaning as a certified contract.

\subsection{Drift Event Classification (Design Constraint DC.2)}

Not all drift crossings are equivalent. We distinguish two categories
based on persistence:

\begin{definition}[Transient Drift Event]
A \emph{transient drift event} occurs when $\Dhat(\tau) \geq \theta$
at a discrete instant but $\Dhat$ returns below $\theta$ within
a persistence window $\Delta T$. The Recovery Loop is the appropriate
response: the system can self-recover without external intervention.
\end{definition}

\begin{definition}[Persistent Drift Event]
\label{def:persistent}
A \emph{persistent drift event} occurs when $\Dhat(t) \geq \theta$
for all $t \in [t_0,\, t_0 + \Delta T]$, where $\Delta T$ is an
operator-defined persistence window specific to the deployment context.
A persistent drift event constitutes a \emph{governance event}: it
exceeds the scope of the execution layer and requires human authority
to resolve, recalibrate baselines, or formally accept the new
behavioral regime.
\end{definition}

\textbf{Design Constraint DC.2:} \emph{The execution layer may respond
to transient drift events autonomously via the Recovery Loop.
Persistent drift events must be escalated to the governance layer;
the execution layer does not have authority to reclassify a persistent
drift event as transient or to accept the new behavioral regime
unilaterally.}

The persistence window $\Delta T$ is operator-defined and
deployment-specific. In our experiments, once drift crossed
$\theta = 0.20$ under monotonic injection, it persisted for the
remainder of the simulation horizon---consistent with a governance-level
event classification under any reasonable $\Delta T$. In production,
$\Delta T$ should reflect the system's recovery latency, escalation
requirements, and domain-specific tolerance for uncertified execution
states.

\textbf{Scope of DC.2.}
DC.2 establishes the \emph{classification} and \emph{routing} of persistent
drift events to the governance layer.
It does not define the governance layer itself.
In particular, the following questions are formally open in this paper:
\begin{itemize}[leftmargin=2em, topsep=2pt]
  \item \textbf{Identity:} which principals constitute the governance layer
        authorized to receive and act on an escalation?
  \item \textbf{Binding:} what mechanism binds a specific human $H_i$ to a
        governance role such that their re-authorization is distinguishable
        from an unauthorized override?
  \item \textbf{Evidence:} what constitutes a valid re-authorization
        record---what information must it capture to support post-hoc
        accountability?
\end{itemize}
These are not implementation details---they are structural prerequisites for
DC.2 to function as a governance mechanism rather than a routing label.
Their formalization is the subject of P8 (Section~\ref{sec:discussion}).

\begin{remark}[HALT Preserves Authority; It Does Not Resolve It]
\label{rem:halt-preserves}
A \HALT\ decision is a safety mechanism, not a resolution mechanism.
When $A(t) = \bot$, \HALT\ correctly prevents execution under undefined
authority---this is the guarantee established by RAM~(P5) and
operationalized by P6.
But \HALT\ answers only one question: \emph{may the system proceed?}
It does not answer the structurally distinct question:
\emph{who may re-authorize execution, and under what conditions?}

Two cases must be distinguished:
\begin{enumerate}[leftmargin=2em, topsep=2pt]
  \item A \HALT\ caused by \emph{unobservable state variables} (the RAM gap,
        P5 Theorem~1) is resolvable by the Recovery Loop.
        The system can self-recover when observability is restored.
  \item A \HALT\ caused by \emph{persistent drift}
        (Definition~\ref{def:persistent}, DC.2) requires an out-of-band
        governance action by an authorized principal.
        The Recovery Loop is not the applicable resolution path:
        the issue is not unobservability---it is a genuine departure from
        the authorized behavioral regime.
        The system cannot and must not self-recover unilaterally.
\end{enumerate}
Conflating these two cases---running the Recovery Loop to exhaustion on a
persistent drift event and then silently resuming---is an authority erosion
event of exactly the type DC.1 is designed to prevent.
The 30.2\% ``stuck'' events in Experiment~4 are empirical evidence of systems
correctly \emph{preserving} authority without \emph{resolving} it.
The formal treatment of who constitutes an authorized principal, what
constitutes valid re-authorization evidence, and how that decision is bound to
an accountable record is deferred to P8.
\end{remark}

\section{Limitations}
\label{sec:limits}

\textbf{MockLLM: controlled methodology, not surrogate.}
All experiments use a deterministic MockLLM by design.
The scientific question this paper answers is not ``do LLMs drift?'' but
``when drift occurs, does the governance stack detect and contain it?''
Answering this requires controlling the drift precisely -- its onset step,
rate, and endpoint -- so that governance behavior can be isolated from
model behavior.
A live LLM would confound both variables: any difference in $T^*$ could
reflect model stochasticity rather than governance signal.
The MockLLM is an ablation instrument, not a limitation.

That said, the governance stack is trace-agnostic -- it observes $\tau$
regardless of what produces it.
Live LLM experiments are the direct next step (Section~\ref{sec:discussion}):
the key question is whether the detection time $T^*$ with natural behavioral
drift is within the $[259, 403]$ range observed here, or whether live drift
is slower (requiring longer run horizons) or faster (requiring lower $\theta$).

\textbf{Depth variance artifact.}
The MockLLM returns depth\,=\,1 for all burn-in steps, giving
$\sigma_\ell = 0$ in $\Azero$.
The fallback normalization ($\max(2\sigma_\ell, 0.5) = 0.5$) slightly
inflates $D_\ell$ for any depth increment above 1.
This is a MockLLM artifact, not a limitation of IML; a live LLM with
natural depth variance would set $\sigma_\ell > 0$ at admission.

\textbf{Stable Evaluation Window.}
The RAM gate's $\IER=0$ guarantee requires that there exists a non-zero
interval during which constructibility holds long enough to complete
evaluation (P6 Assumption~3).
In high-load deployments with rapid state changes, this window may be
too short to guarantee atomic evaluation; we do not model this scenario.

\textbf{Scalability.}
All experiments use a single agent or small $N$ ($\leq 16$).
ACP scales in $O(1)$ per request (hash-table lookup on PatternKey);
IML scales in $O(|\Sigma|)$ per step (distribution computation over tool alphabet);
RAM scales in $O(|D|)$ per request ($D$ = authority-defining components).
Scaling to $N > 1{,}000$ concurrent agents requires validating that
the shared ACP state does not become a bottleneck, which we leave for future work.

\textbf{Scale and heterogeneity.}
Experiments~1, 2, and~4 use a single agent for clarity of isolation.
Experiment~3 uses $N \leq 16$ agents in a controlled round-robin setting.
Real deployments with heterogeneous agents, dynamic task routing,
and long-horizon tasks introduce failure modes beyond this paper's scope:
correlated drift across agents, emergent coordination patterns not
reducible to round-robin interleaving, and recovery loops on shared state.
These are important directions but secondary to the authority binding gap
identified in Section~\ref{sec:discussion}.

\section{Discussion and Future Work}
\label{sec:discussion}

\subsection*{Paper~8: Identity-Bound Governance Under Execution Uncertainty}
\label{sec:p8-scope}

Design Constraints DC.1 and DC.2 define a boundary: the execution layer
cannot cross it, and something must be on the other side.
Paper~8 formalizes what is on the other side.

Its central question: \emph{when $A(t) = \bot$ and the Recovery Loop is
inapplicable (Remark~\ref{rem:halt-preserves}), who may re-authorize
execution, and what evidence must they produce?}

Three formal objects:
\begin{itemize}[leftmargin=2em, topsep=2pt]
  \item \textbf{Principal Set} $\mathcal{P} = \{H_1,\ldots,H_n\}$:
        a pre-registered, identity-bound set of human principals whose
        re-authorization decisions are admissible at the governance layer.
        $\mathcal{P}$ is read-only at the execution layer (a direct
        consequence of DC.1).
  \item \textbf{Authority Resolution Function}
        $G: \mathcal{P} \times \mathcal{S} \times \mathcal{E} \to
        \{\textsc{resume},\,\textsc{deny},\,\textsc{recalibrate}\}$:
        the governance-layer analog of RAM's $F$.
        It constructs authority from human attestation over a defined evidence
        set, not from sensor state.
        Critically, $G$'s \textsc{recalibrate} output is the \emph{only}
        authorized path by which $\Azero$ may change after initialization---
        the out-of-band channel that DC.1 implicitly requires.
  \item \textbf{Accountability Proof Block (APB)}: a two-part record with
        a binding signature, reflecting the division of responsibility
        between system and principal:
        \begin{itemize}[leftmargin=1.5em, topsep=1pt]
          \item $E_s = (\Azero,\;\Dhat(t_e),\;t_e,\;\mathrm{trace},\;
                \mathrm{cause})$ --- the \emph{System Evidence Block},
                constructed by the execution layer from certified runtime
                state. Only the system can produce $E_s$ correctly.
          \item $D_h = (H_i,\;\mathrm{decision},\;\mathrm{rationale},\;
                \mathrm{scope})$ --- the \emph{Human Decision Block},
                added by the acting principal. The principal does not
                restate $E_s$; they declare authority over it.
          \item $\sigma_h = \mathrm{Sign}_{sk_i}(\mathrm{hash}(E_s \,\Vert\,
                D_h))$ --- the principal's signature over the full block.
        \end{itemize}
        The system cannot forge $\sigma_h$ (lacks $sk_i$); the principal
        cannot alter $E_s$ undetected (the signature covers both parts).
        In effect, the principal declares: \emph{``given this certified
        state, I accept responsibility for this decision''} --- preserving
        state integrity while binding governance authority to a specific
        identity. The APB extends the P6 audit artifact: $E_s$ maps to
        P6's existing fields; $D_h$ and $\sigma_h$ are the P8 additions.
\end{itemize}

Three theorems P8 will prove:
(1)~\emph{Governance Completeness}---every HALT event resolves through either
the Recovery Loop or a signed APB; no third path exists.
(2)~\emph{Non-Repudiability}---no principal can deny a governance act for
which a valid APB exists.
(3)~\emph{Impossibility of Anonymous Re-Authorization}---the execution layer
cannot generate a valid APB without the principal's private credential.

The 167 ESCALATE and 248 ``stuck'' events from Experiment~4 constitute the
empirical evidence base for P8's validation experiments.

\textbf{MCP/A2A integration.}
The governance stack as implemented operates on tool-call traces from a
LangGraph StateGraph. The Model Context Protocol (MCP) and Agent-to-Agent
(A2A) communication protocols define standard interfaces for tool invocation
that the ACP gate and RAM check could intercept natively.
Instrumentation at the MCP server layer would allow the stack to govern
any MCP-compatible agent without modifying the agent's code.

\textbf{Live LLM experiments.}
The next step is to run the same four experiments with a live LLM
(Claude, GPT-4o, or an open-source model via Ollama) and measure how
natural behavioral drift compares to the MockLLM's induced drift.
The key question is whether IML's $T^*$ in a live setting matches the
expected range, and whether the RAM HALT rate at 70\% coverage is
similar or higher (live LLMs may produce more uncertain tool invocations).

\textbf{Redis/distributed ACP backend.}
The current implementation uses an in-memory ACP gate.
P1 defines a Redis-based implementation for distributed deployments.
Validating that the evaluate-then-mutate contract holds under concurrent
writes (requiring atomic read-modify-write at the Redis level) is
a prerequisite for production deployment.

\textbf{Adaptive coverage.}
The RAM gate uses a fixed coverage level.
An adaptive policy -- reducing the HALT threshold when IML drift is low,
and tightening it as $\Dhat$ approaches the boundary -- would reduce OCR
without sacrificing the $\IER=0$ guarantee.
This is the natural integration point for the IML--RAM feedback loop.

\textbf{Design trade-offs: decision integrity over execution continuity.}
The architecture makes an explicit choice: under uncertified state
conditions, decision integrity takes precedence over execution continuity.
This reflects the classical safety/liveness tension in distributed
systems~\cite{lamport2001paxos,alpern1985defining}: a system can guarantee
either that bad things never happen (safety) or that good things eventually
happen (liveness), but not both unconditionally under failure.
A system that adapts its own governance parameters under runtime pressure
may preserve availability---but does so by silently degrading authority.
We term this \emph{controlled brittleness}: the stack fails closed rather
than adapts, precisely because adaptation under uncertified conditions
cannot be distinguished from authority erosion.
The cost is measurable---OCR quantifies it directly---and the benefit is
that every execution decision remains bound to a certified governance state.
This trade-off is most visible in the HALT events of Experiment~4: each
represents the stack choosing integrity over continuity. That is not a
defect; it is the architecture functioning as designed.
Design Constraints DC.1 and DC.2 make this choice explicit and
non-negotiable at the execution layer: recalibration and reclassification
are governance actions, not execution actions.

\section{Conclusion}
\label{sec:conclusion}

We have presented the first empirical validation of the Agent Governance
Stack (Papers~0--6, as specified in the Zenodo preprints cited herein)
applied to real LangGraph agent workflows.
Four experiments confirm the four core theoretical predictions of the series:
IML detects compliant drift invisible to enforcement (Exp.\,1);
RAM achieves $\IER=0$ at all state-coverage levels (Exp.\,2);
ACP bounds the multi-agent coordination window at $\CW_{\mathrm{appr}} = 2N$
with zero deviation (Exp.\,3);
and the integrated stack converges without deadlock and maintains
conditional liveness (Exp.\,4).

Beyond confirmation, the implementation surfaces three empirical refinements
that sharpen the theoretical framework---the ACP baseline-RS assumption,
the liveness-rate classification for the conditional liveness theorem, and
the EMA-parameter dependency in the convergence bound---and establishes
two design constraints that make explicit the structural limits of the
execution layer: Separation of Recalibration Authority (DC.1) and
Drift Event Classification (DC.2).
These contributions do not invalidate the existing proofs; they make
implicit assumptions explicit, add parametric precision to asymptotic
claims, and formalize the architectural boundary between what the
execution layer may decide autonomously and what requires governance
authority.

The governance stack is publicly available as an open-source Python library
at \url{https://github.com/chelof100/agent-governance-applied}, with
all experimental data, seeds, and results included for full reproducibility.
Archived version with DOI: \url{https://doi.org/10.5281/zenodo.19929771}.
